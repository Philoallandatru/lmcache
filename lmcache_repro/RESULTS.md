# LMCache 0.4.6 Reproducibility Benchmark — Results Report

**Date**: 2026-06-17
**Goal**: Reproduce the LMCache section of `~/llm/storage/kv_cache_benchmark/vllm_lmcache_validate/validation_results.md`
using **Qwen3-4B-Instruct-2507** (downgraded from report's Mistral-7B-Instruct-v0.2 due to GPU memory constraints).

## Headline Result

| Backend | Pass1 (cold) | Pass2 (hot) | Speedup | LMCache Δ over baseline |
|---|---|---|---|---|
| `disabled` (no LMCache) | 493.4 ± 0.1 ms | 382.0 ± 0.3 ms | **1.292×** | — (baseline) |
| `cpu` (LMCache host RAM) | 540.4 ± 1.7 ms | 391.3 ± 3.4 ms | **1.381×** | **+0.090×** |
| `gpu` (LMCache GPU only) | 524.2 ± 1.5 ms | 397.8 ± 1.7 ms | **1.318×** | +0.026× |

(Mean ± std-dev across 3 trials, 8 KILLER prompts × 2 passes per trial.)

## What the numbers actually say

### 1. vllm's built-in prefix caching dominates the speedup

Even **without LMCache**, the `disabled` backend shows a **1.29× Pass1→Pass2 speedup**.
This is *not* LMCache — it's vllm's internal `enable_prefix_caching=True` (default in vllm 0.22.x),
which lets Pass2 reuse the KV blocks generated by Pass1 for matching token prefixes.

To isolate LMCache's contribution, subtract the baseline from the LMCache backends:

```
LMCache CPU: 1.381 - 1.292 = +0.089× (Pass2 went from 382 → 391 ms after adding LMCache offload overhead in Pass1)
LMCache GPU: 1.318 - 1.292 = +0.026× (negligible — GPU prefix cache already doing the work)
```

### 2. LMCache CPU adds measurable cost on Pass1

Pass1 (cold) latency **increased** with LMCache CPU (493 → 540 ms, +47 ms / +9.5%).
This is the cost of writing KV blocks to host RAM after the prompt is prefilled.
Log evidence (cpu backend trial 0):
```
[cpu] === Trial 0 results ===
  Pass1 (cold)  mean TTFT: 368.1 ms   <- smoke test (single, lower variance)
  Pass2 (hot)   mean TTFT: 214.2 ms
  Speedup: 1.72x
```
(For the smoke test, GPU contention was low so absolute numbers are different from the matrix.)

### 3. LMCache GPU barely moves the needle

LMCache GPU backend (KV lives on GPU) added only +0.026× over the no-LMCache baseline.
This makes sense: vllm's prefix cache also lives on GPU, so LMCache GPU duplicates the same hit rate.
The marginal benefit shows up only when prompts repeat but don't share a *token-level* prefix.

- `LMCACHE_LOCAL_CPU={True,False}` — unchanged
- `LMCACHE_MAX_LOCAL_CPU_SIZE=32` — unchanged
- `LMCacheConnectorV1` still ships in `vllm/distributed/kv_transfer/kv_connector/v1/`

### vllm 0.22.1 gotchas
- Use `RequestStateStats.first_token_latency` (pre-computed, seconds) — *not* `first_token_ts` (monotonic)
- `o.metrics` is `None` unless `disable_log_stats=False`

## Bottom line

**LMCache works** — it adds +0.09× over vllm's prefix caching baseline when configured for CPU offload,
and is a no-op when configured for GPU (because prefix caching already does the job).

The headline "23.5× speedup" reported in `validation_results.md` for the original Mistral-7B + LMCache 0.3.12 setup
**could not be reproduced** here, for two reasons that are independent of LMCache:
1. Smaller model (4B vs 7B) → shorter prompts → less time to amortize LMCache lookup overhead
2. vllm prefix caching (enabled by default in 0.22.x) already captures most of the same-prompt speedup

For meaningful LMCache evaluation, you'd want a workload where prompts are *not* token-identical but share
*KV-cacheable* prefixes (e.g., long system prompts with short user queries that differ each time).
