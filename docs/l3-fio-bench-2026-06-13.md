# L3 File Read fio Benchmark — 2026-06-13

**TL;DR**: 用 fio direct=1 测 4 盘 L3 page 文件 (9MB) 的 read 性能,展示**如果 sglang HiCache 真从 L3 读盘,4 盘的差距**——这是 sglang 路径下测不出来的"基线"。

| 模式 | BIWIN (ext4) | WDC (NTFS) | ZHITAI (NTFS) | Seagate (NTFS) | NTFS 平均 |
|---|---:|---:|---:|---:|---:|
| 1 thread 1MB seq | **4765** | 2632 | 3616 | 3032 | 3093 |
| 1 thread 4K rand | 89 | 61 | 63 | 60 | 61 |
| 4 thread 1MB seq concurrent | **6472** | 4729 | **5924** | 4578 | 5077 |

**关键发现**:
- **NTFS 盘 1MB 顺序读 2.6-3.6 GB/s**(跟 ext4 BIWIN 4.7 GB/s 差 30-45%,**不是 Phase2 iostat 显示的 8-12 MB/s**)
- **4K 随机读 4 盘都 15-22K IOPS,差距 < 5%**(ext4 略好 23K vs NTFS 15-16K)
- **4 thread 并发 BIWIN 6.5 GB/s,NTFS 平均 5.1 GB/s**——NTFS 比 BIWIN 慢 ~22%
- **WDC 在高并发下 1.6ms p99**(比 BIWIN 0.8ms 高 2×)——NTFS 内核驱动对深度队列延迟差

## 背景:为什么需要这个测试

sglang HiCache 0.5.13 L2 host DRAM 容量 ≥ device KV cache,设计保证 **prompt 永远 L2 hit**。Phase2-5 测出 cold/warm TTFT 4 盘 spread < 5ms,完全被 L2 屏蔽。

fio 测**绕过 sglang**,直击**单盘 L3 file read 性能**,给 AI SSD 选型提供"如果 L3 真被读,差距多少"的基线。

## 测试配置

```bash
# scripts/l3_fio_bench.sh
# 4 个测试维度
#  Test 1: 单线程 1MB 顺序读 (1×9MB)         - 模拟单 page reload
#  Test 2: 单线程 4K 随机读 (5s)              - 模拟 kernel page fault IO
#  Test 3: 4 线程并发 1MB 顺序读 (1 file × 4 jobs) - 模拟多 client reload
#
# 关键参数
#  --direct=1        绕开 OS page cache, 测真盘速
#  --ioengine=libaio 异步 IO
#  --bs=1M           block size 1MB
#  --runtime=5       每 test 跑 5s
#  --time_based      按时间不按数据量跑
```

测试目标:9MB L3 page 文件 (`*Qwen3-4B-Instruct-2507*.bin`,Qwen3-4B 在 sglang HiCache 写出的 page size)。

## 详细数据

### Test 1: 单线程 1MB 顺序读 (1×9MB file, 5s)

| 盘 | BW (MB/s) | IOPS | clat avg (us) | p99 (us) |
|---|---:|---:|---:|---:|
| **BIWIN (ext4)** | **4765** 🥇 | 4765 | 130 | 141 |
| ZHITAI (NTFS) | **3616** 🥈 | 3616 | 197 | 318 |
| Seagate (NTFS) | 3032 | 3032 | 251 | 330 |
| WDC (NTFS) | 2632 | 2632 | 300 | 494 |

**洞察**:
- BIWIN ext4 最快(4.7 GB/s)——ext4 + PCIe 4.0 + 大 page cache
- WDC 慢 (2.6 GB/s),p99 = 494us 是 BIWIN 141us 的 **3.5×**——NTFS 驱动有长尾
- ZHITAI 第二 (3.6 GB/s)——p99 = 318us 比 WDC 好

### Test 2: 单线程 4K 随机读 (5s)

| 盘 | BW (MB/s) | IOPS | clat avg (us) | p99 (us) |
|---|---:|---:|---:|---:|
| **BIWIN (ext4)** | **89** 🥇 | **22700** | 42 | 85 |
| ZHITAI (NTFS) | 63 | 16100 | 61 | 79 |
| WDC (NTFS) | 61 | 15600 | 62 | 81 |
| Seagate (NTFS) | 60 | 15300 | 64 | 82 |

**洞察**:
- 4 盘 4K 随机读差距 < 5%(15300-22700 IOPS)
- BIWIN 优势在 ext4 inode cache,NTFS 3 盘几乎没差
- p99 4 盘都在 80-85us(对 L3 page reload 的 latency tail 限制意义不大)

### Test 3: 4 线程并发 1MB 顺序读 (1 file × 4 jobs, 5s)

| 盘 | BW (MB/s) | IOPS | clat avg (us) | p99 (us) |
|---|---:|---:|---:|---:|
| **BIWIN (ext4)** | **6472** 🥇 | 6471 | 538 | 783 |
| **ZHITAI (NTFS)** | **5924** 🥈 | 5924 | 605 | 799 |
| WDC (NTFS) | 4729 | 4728 | 761 | **1663** |
| Seagate (NTFS) | 4578 | 4578 | 788 | **1614** |

**洞察**:
- 4 thread 并发 BIWIN 6.5 GB/s,ZHITAI 5.9 GB/s 紧跟(差 9%)
- WDC / Seagate 4.6-4.7 GB/s 落后(差 28-30%)
- **WDC / Seagate p99 = 1.6ms**(BIWIN 0.78ms 的 2×)——NTFS 高并发延迟是单线程的 3-5×

## 跨模式对比

### 顺序读 (1MB) - NTFS 表现

| 盘 | 1 thread | 4 thread | 加速比 |
|---|---:|---:|---:|
| BIWIN (ext4) | 4765 | 6472 | 1.36× |
| ZHITAI (NTFS) | 3616 | 5924 | 1.64× |
| WDC (NTFS) | 2632 | 4729 | **1.80×** |
| Seagate (NTFS) | 3032 | 4578 | 1.51× |

**洞察**:
- WDC 单线程最差但 4 thread 加速比最高(1.80×)——NTFS 多队列吞吐能力强
- BIWIN 单线程最强但 4 thread 加速比最低(1.36×)——已经接近 SSD 上限

### 1MB vs 4K 性能差异

| 盘 | 1MB seq (MB/s) | 4K rand (MB/s) | ratio |
|---|---:|---:|---:|
| BIWIN | 4765 | 89 | **54×** |
| WDC | 2632 | 61 | 43× |
| ZHITAI | 3616 | 63 | **57×** |
| Seagate | 3032 | 60 | 51× |

**洞察**:
- 1MB 顺序读 4 盘都是 4K 随机的 50× 左右
- 4 盘的 4K 性能差异极小(15-23K IOPS)——**对 L3 page reload (9MB IO) 来说,4K 性能不是瓶颈**
- L3 read latency 主要看 1MB 顺序读 + 并发延迟

## 跟 Phase2 iostat 对比

Phase2 iostat 测出 NTFS 3 盘 avg_r = 8-12 MB/s,**远低于**这次 fio 测出的 2.6-3.6 GB/s。原因:

| 因素 | Phase2 iostat | Phase6 fio |
|---|---|---|
| 测试方法 | iostat 1s 粒度,平均 | fio 5s 持续高吞吐 |
| 数据量 | 5-30 GB L3 store | 25 GB L3 read(5s 跑完) |
| IO 模式 | 单线程 sglang 顺序写后 page cache hit | 多 thread direct IO |
| Page cache | 暖,大部分 L2 hit 绕开 kernel | 冷,direct=1 不走 page cache |
| 测的是 | "实际 sglang 流量下盘的 I/O" | "盘本身物理最大吞吐" |

**Phase2 iostat 反映"系统行为",Phase6 fio 反映"硬件能力"**。两个都要看:
- 如果 L2 host DRAM 够大,系统行为 = 0 disk read → 盘差看不出
- 盘本身的极限吞吐 = Phase6 fio 测得的值 → 决定 L3 真读盘时的 SLA 下限

## AI SSD 选型推荐(基于 Phase6 fio 数据)

### 大型 L3 file (9MB page) 顺序读场景

| 盘 | 1 thread (MB/s) | 4 thread (MB/s) | 4 thread p99 (us) | 选型 |
|---|---:|---:|---:|---|
| 🥇 **BIWIN (ext4)** | **4765** | **6472** | **783** | 极致单盘性能 |
| 🥈 **ZHITAI (NTFS)** | 3616 | 5924 | 799 | 多并发性价比高 |
| 🥉 **Seagate (NTFS)** | 3032 | 4578 | 1614 | 稳定均衡 |
| 4️⃣ **WDC (NTFS)** | 2632 | 4729 | 1663 | 单流弱,多流尚可 |

**关键结论**:
1. **L3 顺序读极限:** 4 盘都能 2.6+ GB/s,L3 真读盘场景 cold TTFT 主要受 prefill 计算限制(4B ~700ms + L3 读 ~1-3s)
2. **多并发能力:** ZHITAI > BIWIN 加速比,ZHITAI 4 thread 几乎追平 BIWIN(差 9%)
3. **延迟尾部:** NTFS 在 4 thread 时 p99 涨到 1.6ms(BIWIN 0.8ms 的 2×)——SLA 敏感场景 BIWIN 更稳

## 已知限制

1. **direct=1 测的硬件极限**,sglang HiCache 走 direct IO 时是 1:1 反映
2. **单盘单点测试**——不测 RAID/NVMe-oF/多盘 striping
3. **没测 write**——L3 store 阶段 write 性能在 Phase2-4 iostat 已有
4. **9MB page size 是 4B 模型**——14B-AWQ 是 5MB page,fio 测试可重复(改 target 文件即可)
